## gh CLI Extension Packaging Guide

### 1. Naming Requirements (Mandatory, Exact)

**Repository name:** Must start with `gh-`. The remainder becomes the extension command name.
- Example: repo `gh-actions-audit` --> user runs `gh actions-audit`

**Executable file:** Must be at the repo root with the **exact same name** as the repository (no `.sh` extension).
- For your case: the file must be named `gh-actions-audit` (not `gh-actions-audit.sh`)
- Must be executable (`chmod +x gh-actions-audit`)
- Shebang must be `#!/usr/bin/env bash`

**What this means for your repo:** The current repo is named `gh-action-security-audit` and the script is `gh-actions-audit.sh`. You have two options:
- (A) Rename the repo to `gh-actions-audit` and rename the script from `gh-actions-audit.sh` to `gh-actions-audit` (dropping `.sh`), or
- (B) Keep the current repo name `gh-action-security-audit` and rename the script to `gh-action-security-audit` -- but then the command becomes `gh action-security-audit`.

Choose whichever command name you prefer users to type.

### 2. Manifest File: Not Needed for Bash Extensions

The `manifest.yml` file is an **internal implementation detail** of gh's extension manager. It is automatically generated by gh when it installs a **precompiled binary** extension (to track which OS/arch binary was downloaded, the version, etc.). You do **not** create it yourself.

For a **bash script extension**, there is no manifest file at all. The extension is just:
- A git repository named `gh-<name>`
- An executable file named `gh-<name>` at the repo root

That is the entire contract. No `package.json`, no `manifest.yml`, no `extension.yml`.

### 3. Shell-Based vs Go-Based Extensions

| Aspect | Shell/Bash Extension | Go (Precompiled) Extension |
|---|---|---|
| **Installation** | `gh extension install` clones the repo as a symlink | Downloads a platform-specific binary from a GitHub Release |
| **File needed** | Single executable `gh-<name>` at repo root | Release assets named `gh-<name>-<OS>-<ARCH>[.exe]` |
| **Dependencies** | User must have bash (+ any tools the script calls like `gh`, `jq`, etc.) | Self-contained binary, no runtime deps |
| **Updates** | `gh extension upgrade` does a `git pull` | Downloads new binary from latest release |
| **Build tooling** | None needed | Use `cli/gh-extension-precompile` GitHub Action to automate cross-compilation |
| **manifest.yml** | Not generated | Auto-generated on install to track binary metadata |

**For your case:** Bash is the right choice. Your script already targets bash 3.2+ and only depends on `gh` CLI and standard Unix tools -- both of which are guaranteed to be present on any system where someone would install a gh extension.

### 4. Installation and Upgrade Flow

**Install from remote (end user):**
```bash
gh extension install rapidfort/gh-actions-audit
# Now the user can run:
gh actions-audit my-org
gh actions-audit my-org --out report.md --csv report.csv
```

**Install from local directory (development):**
```bash
cd gh-actions-audit/
gh extension install .
# Creates a symlink -- edits to the script take effect immediately
```

**Upgrade:**
```bash
gh extension upgrade actions-audit    # single extension
gh extension upgrade --all            # all installed extensions
```

**Remove:**
```bash
gh extension remove actions-audit
```

**List installed:**
```bash
gh extension list
```

**Discovery:**
```bash
gh extension search actions-audit
gh extension browse    # interactive TUI sorted by stars
```

### 5. Conventions for --help, --version, Output Formatting

There are **no enforced conventions** from gh itself -- it just passes all arguments through to your script. However, these are the de facto standards from popular extensions:

**--help / -h:** Strongly expected. Your script already has a usage block in comments; you should also handle `--help` and `-h` flags to print it. gh does **not** auto-generate help for extensions.

**--version / -V:** Optional but good practice. Users expect it. Do not use `-v` for version (conventionally that means verbose). Use `-V` or `--version`.

**Output formatting:** There are no strict rules, but:
- stdout for primary output (reports, data)
- stderr for progress/status messages
- ANSI colors auto-disabled when not a TTY (you already do this)
- Exit code 0 for success, non-zero for errors
- The `gh` ecosystem conventions favor machine-readable output options (`--json`, `--jq`), but for an audit tool, markdown + CSV is perfectly fine

### 6. How Real Bash-Based gh Extensions Are Structured

**meiji163/gh-notify** (324+ stars) -- the most popular pure-bash gh extension:
```
gh-notify/
  gh-notify          # The executable bash script (no .sh extension)
  readme.md          # Documentation
  LICENSE            # License file
```

That is literally it. No build system, no manifest, no config files. The executable at the root is the entire extension.

**mislav/gh-branch** -- another popular bash extension:
```
gh-branch/
  gh-branch          # Executable bash script
  README.md
```

**Pattern across all bash extensions:**
- Root-level executable named identically to the repo
- README with install instructions: `gh extension install <owner>/gh-<name>`
- Optional: LICENSE, .github/workflows for CI
- No `.sh` extension on the executable

### 7. Publishing / Registry / Discoverability

There is **no formal registry or marketplace** with a submission process. Discovery is entirely topic-based:

1. **Add the `gh-extension` topic** to your GitHub repository. This is the single most important step. Without it, the extension will not appear in `gh extension search` or `gh extension browse`.

2. **That's it.** There is no approval process, no review, no publishing step. The moment your repo is public, has the correct naming, and has the `gh-extension` topic, it is discoverable.

3. Extensions are **browseable** via:
   - `gh extension browse` (interactive TUI in terminal, sorted by stars)
   - `gh extension search <keyword>` (CLI search)
   - https://github.com/topics/gh-extension (web)
