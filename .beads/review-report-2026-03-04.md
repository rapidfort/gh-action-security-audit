## P0 — Critical Detection Gaps

### 1. Expression injection in `run:` blocks
- **Reviewers:** All 3 + research
- **Card:** exh
- `${{ github.event.pull_request.title }}`, `${{ github.head_ref }}`, etc. interpolated directly into `run:` blocks
- **Real-world exploits:** Ultralytics YOLO (branch name injection), Nx (PR title injection), hackerbot-claw
- README lists this as attack vector #3 but **the tool doesn't detect it**
- Dangerous contexts: `issue.title`, `issue.body`, `pull_request.title`, `pull_request.body`, `pull_request.head.ref`, `comment.body`, `review.body`, `commits[*].message`, `head_ref`
- **Implementation:** Add `classify_expr_injection()` function following `classify_prt`/`classify_ic` pattern
- **Difficulty:** Medium (multi-line `run: |` blocks need awk or careful grep)

### 2. Unpinned action references (tag vs SHA)
- **Reviewers:** Detection + research
- **Card:** e2w
- `uses: actions/checkout@v4` vs `uses: actions/checkout@692973e3d937129bcbf40652eb9f2f61becf3332`
- **Real-world exploit:** tj-actions/changed-files compromise (23K+ repos, CVE-2025-30066)
- Tags can be retroactively moved to point to malicious commits; SHA pins are immutable
- GitHub added SHA pinning enforcement via org policy (Aug 2025)
- **Implementation:** `grep -E 'uses:.*@' | grep -vE '@[0-9a-f]{40}'` — simple single grep per file
- Report as ratio per repo (e.g., "2/15 SHA-pinned")
- **Difficulty:** Easy

## P1 — High Priority

### 3. `workflow_run` trigger detection
- **Reviewers:** Detection + research
- **Card:** 8yr
- `workflow_run` bypasses fork PR restrictions — runs with write permissions and secrets even when triggering workflow was an unprivileged fork PR
- Combined with `download-artifact` = artifact poisoning vector (Ultralytics YOLO attack)
- Sub-classify: `workflow_run` + artifact download = HIGH, + checkout = MEDIUM, alone = LOW
- **Implementation:** Add `classify_wfr()` function following `classify_prt` pattern
- **Difficulty:** Easy-Medium

### 4. Self-hosted runner exposure
- **Reviewers:** Detection + research
- **Card:** 6c6
- Self-hosted runners are persistent (not ephemeral VMs like GitHub-hosted)
- Compromised runner gives attacker host network access, cached credentials, lateral movement
- Critical when combined with `pull_request_target`, `issue_comment`, or `workflow_run`
- **Implementation:** `grep -qEi 'runs-on:.*self-hosted'` — single grep
- **Difficulty:** Easy

### 5. Dangerous `permissions:` values
- **Reviewers:** Detection
- **Card:** zys
- Currently checks whether `permissions:` block EXISTS but not WHAT IT GRANTS
- `permissions: write-all` is just as dangerous as no permissions block
- Detect: `write-all`, `contents: write`, `actions: write`, `packages: write`, `id-token: write`
- `id-token: write` + `pull_request_target` = OIDC token theft risk
- **Implementation:** Additional grep patterns in `analyze_repo`
- **Difficulty:** Easy

### 6. Silent API failure handling
- **Reviewers:** Quality + QA
- **Card:** ct5
- Line 282: `gh repo list` failure silently swallowed by `2>/dev/null`
- If API fails (network drop, auth expiry, rate limit), script produces empty report with no warning
- **This is a security tool — missing data is worse than a crash**
- Also: Phase 1 download loop `|| true` on every API call means partial downloads are silent
- **Implementation:** Add `|| { crit "Failed to list repos"; exit 1; }` and track download failure count
- **Difficulty:** Easy

### 7. Duplicate Phase 4 API call
- **Reviewers:** Quality + QA
- **Card:** i45
- Lines 597-604: Same endpoint `orgs/$ORG/actions/permissions/workflow` called twice with different `--jq`
- Wastes API call, doubles rate-limit cost, introduces inconsistency if one fails
- Also: Dead code — `"True"` (capital T) check on line 645 is unreachable (jq returns lowercase)
- **Implementation:** Fetch once, extract both fields from same response
- **Difficulty:** Easy

### 8. Summary statistics section

- Per-repo table is hard to triage for executives/security leads without headline numbers

### 14. Use `grep -F` for secret name lookup in `SECRET_MAP_FILE`
- **Card:** hxf
- Line 570: `grep "^${secret_name}|"` uses regex matching
- GitHub secret names are `[A-Za-z_][A-Za-z0-9_]*` so safe in practice, but defense in depth
- Alternative: `awk -F'|' -v name="$secret_name" '$1 == name {print $2}'`
- **Difficulty:** Easy

### 15. Refactor `analyze_repo` output to eliminate fragile `head`/`tail` split
- **Card:** 1rx
- `analyze_repo` outputs 2 lines split by `head -1` / `tail -1` — breaks if any cell has newlines
- Option 1: Pass temp file paths as args, write directly (cleanest)
- Option 2: Validate line count before splitting
- **Difficulty:** Easy
